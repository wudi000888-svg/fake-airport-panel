# v2.1.0 SQLite-only 技术债清理设计

## 背景

v2.0.x 已经默认启用 SQLite，但仍保留旧 JSON 运行时 fallback、JSON 到 SQLite 迁移脚本、SQLite 导出回 JSON 脚本，以及多处“JSON 兼容”文案。继续保留双存储会让用户、套餐、订单、支付、节点、注册、备份和安装脚本都长期背负两套路径，容易出现测试污染、升级脚本复杂和线上数据来源不一致。

v2.1.0 的目标是把 fake-ui 收敛为 SQLite-only 业务数据模型。旧 JSON 文件不再作为运行时数据源。旧版本用户必须先升级到 v2.0.1，并在 v2.0.1 环境中完成数据整理后，再升级到 v2.1.0。

## 范围

| 项目 | 决策 |
|---|---|
| 业务数据 | 用户、套餐、订单、支付、节点、注册/找回、管理员 profile、link settings 全部走 SQLite |
| JSON runtime fallback | 移除 `FAKE_UI_STORE` 切换和 plans/orders/payments JSON fallback |
| 一次性旧 JSON 迁移 | v2.1.0 不内置旧 JSON 迁移入口；升级文档要求先到 v2.0.1 |
| HTTP JSON | 保留，API 仍返回 JSON |
| Xray 配置 | 保留 `data/xray/config.json`，这是 Xray 原生配置，不属于旧业务存储 |
| Hysteria2 配置 | 保留 `server.yaml` |
| SQLite `data_json` 字段 | 保留，用作表结构演进缓冲，不等同于旧 JSON 文件运行时兼容 |
| 备份 | 只备份 SQLite DB、WAL/SHM、auth/登录文本、审计/订阅日志和协议配置 |

## 架构

SQLite schema 已包含 `users`、`nodes`、`registrations`、`password_resets`、`settings` 等表，但 v2.0.x 只真正使用了 plans/orders/payments 仓储。v2.1.0 会补齐剩余仓储：

| 仓储 | 表 |
|---|---|
| `SQLiteUsersRepository` | `users` |
| `SQLiteNodesRepository` | `nodes` |
| `SQLiteRegistrationsRepository` | `registrations`、`password_resets` |
| `SQLiteSettingsRepository` | `settings`，复用支付 settings 仓储或迁到通用 settings 仓储 |

业务 store 文件保留现有函数名，减少 API 层改动；内部实现改为直接 `db_schema.migrate()` 后访问 SQLite。`json_store.py` 只允许用于 auth、backup metadata、demo/test 辅助或后续完全删除前的非业务文件，不再被业务 store 作为主存储依赖。

## 升级策略

v2.1.0 只支持从 v2.0.1 升级。升级前要求：

1. 运行 v2.0.1。
2. 确认 `data/panel/fake-ui.db` 存在且业务数据已写入 SQLite。
3. 备份整个 `/opt/fake-airport`。
4. 再部署 v2.1.0。

v2.1.0 安装脚本全新安装时直接初始化 SQLite 和必要 settings，不再生成业务 JSON 文件后迁移。

## 测试

| 层级 | 验证 |
|---|---|
| 单元测试 | SQLite 用户、节点、注册、settings 仓储 CRUD |
| Store 测试 | 不设置 `FAKE_UI_STORE` 时也只写 SQLite，不生成业务 JSON 文件 |
| API 测试 | 用户、节点、套餐、订单、支付、注册、备份路径不回归 |
| 脚本测试 | 安装脚本不再引用 `migrate-json-to-sqlite.py` 或 `export-sqlite-to-json.py` |
| 远端预检 | 新加坡临时目录 `bash -n`、`docker compose config`、Python 编译、pytest |

## 非目标

| 非目标 | 原因 |
|---|---|
| 多服务器控制面 | 属于 v3 方向 |
| 删除 Xray `config.json` | 协议配置原生格式，不是旧业务存储 |
| 删除所有 Python `json` import | HTTP/API/RPC 仍需要 JSON |
| 直接兼容 v1/v1.2 JSON 升级 | v2.1.0 明确要求先升级到 v2.0.1 |
